Workflow

From request to paid proof in three steps.

The product keeps the workflow simple: create the invoice, collect through the right rail, then turn the payment into a portable proof record.

1
Invoice

Create a clean payment request.

Enter client, amount, due date, and payment rail. HermesWork creates the record and exposes Stripe/x402 paths from the same invoice.

2
Collect

Track status without guessing.

Paid, pending, and overdue work is visible from the dashboard, with client-level signals for payment speed and relationship health.

3
Prove

Attach proof to completed work.

After payment confirmation, the system stores a credential-style record with amount, client, rail, timestamp, and transaction reference.

FAQ

Practical answers.

Is this production-ready?

The backend is hardened for a demo: API-key protected writes, rate limits, safe rendering, and mock fallbacks. For production, add persistent storage and real Stripe webhook secrets.

Why does the backend sleep?

Render free web services sleep after inactivity. Visit the backend health URL before a demo to wake it up. The frontend static site stays available.

Do I need real Stripe keys?

No for the demo. Use mock values to validate the UI, then replace them with real test keys and webhook secrets when needed.

What should I show judges?

Show the landing page, open the dashboard, save the API key, seed demo data, create an invoice, and confirm a payment-backed credential record.
